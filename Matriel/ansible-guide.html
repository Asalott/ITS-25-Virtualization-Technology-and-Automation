01 — Overview

What is Ansible?

Ansible is an open-source IT automation engine built by Red Hat. It lets you define your infrastructure in simple YAML files — no special agents needed on target machines, just SSH (or WinRM for Windows).

⚡

Agentless

Uses SSH to connect. No daemon or agent software to install on managed nodes.

📄

Declarative YAML

Playbooks describe the desired state in human-readable YAML syntax.

🔁

Idempotent

Running the same playbook multiple times produces the same result — safe to re-run.

🧩

Huge Module Library

5,000+ built-in modules for cloud, networking, databases, containers, and more.

How it works

The Control Node (your machine or a CI server) runs Ansible. It reads your inventory (list of servers), compiles tasks from playbooks, and pushes them over SSH to Managed Nodes. No software needs to be pre-installed — just Python on the remote host.

Architecture 
  ┌─────────────────┐       SSH / WinRM        ┌──────────────┐
  │  Control Node   │ ────────────────────────▶ │ Managed Node │
  │  (your laptop / │                           │  web-01      │
  │   CI server)    │ ────────────────────────▶ │ Managed Node │
  │                 │                           │  db-01       │
  │  ansible        │ ────────────────────────▶ │ Managed Node │
  │  playbooks      │                           │  cache-01    │
  └─────────────────┘                           └──────────────┘
         │
  reads: inventory.ini / hosts.yml
         playbooks/*.yml
         roles/

02 — Getting Started

Installation & Setup

Ubuntu / Debian 
# Add official PPA
sudo apt update
sudo apt install -y software-properties-common
sudo add-apt-repository --yes --update ppa:ansible/ansible
sudo apt install -y ansible

# Verify
ansible --version
macOS (Homebrew) 
brew install ansible
ansible --version
⚠️ Windows Control Node Ansible doesn't run natively on Windows as a control node. Use WSL2 (Ubuntu) and follow the Linux instructions inside it.
PowerShell (enable WSL2) 
wsl --install -d Ubuntu
# Then open Ubuntu terminal and follow Linux steps
pip (any OS with Python 3.9+) 
python3 -m pip install --user ansible
# Or inside a virtualenv (recommended)
python3 -m venv .venv
source .venv/bin/activate
pip install ansible
ansible --version

First ping test

1

Create a minimal inventory

hosts.ini
[webservers]
192.168.1.10  ansible_user=ubuntu
192.168.1.11  ansible_user=ubuntu
2

Run an ad-hoc ping

terminal
ansible webservers -i hosts.ini -m ping

You should see pong returned from each host.

3

Run your first playbook

terminal
ansible-playbook -i hosts.ini site.yml
03 — Project Layout

File Structure

Ansible doesn't enforce a strict layout, but following the official best-practice structure keeps large projects manageable.

my-ansible-project/ ├── ansible.cfg # project-level config (overrides /etc/ansible/ansible.cfg) ├── hosts.ini # OR inventory/ directory │ ├── site.yml # master playbook — imports everything ├── webservers.yml # playbook for web tier ├── databases.yml # playbook for db tier │ ├── inventory/ │ ├── production # production hosts │ ├── staging # staging hosts │ └── group_vars/ │ ├── all.yml # vars for ALL hosts │ ├── webservers.yml # vars for webservers group │ └── databases.yml │ ├── group_vars/ # global group variables │ └── all.yml │ ├── host_vars/ # per-host variables │ └── web-01.yml │ ├── roles/ │ ├── common/ # applied to all hosts │ │ ├── tasks/ │ │ │ └── main.yml │ │ ├── handlers/ │ │ │ └── main.yml │ │ ├── templates/ │ │ │ └── ntp.conf.j2 │ │ ├── files/ │ │ ├── vars/ │ │ │ └── main.yml │ │ ├── defaults/ │ │ │ └── main.yml # lowest-priority defaults │ │ └── meta/ │ │ └── main.yml # role dependencies │ └── nginx/ # another role │ └── library/ # custom modules
ansible.cfg
[defaults]
inventory          = ./inventory/production
remote_user        = ubuntu
private_key_file   = ~/.ssh/id_rsa
host_key_checking  = False
roles_path         = ./roles
retry_files_enabled= False
stdout_callback    = yaml

[privilege_escalation]
become        = True
become_method = sudo
become_user   = root

[ssh_connection]
pipelining    = True
ssh_args      = -o ControlMaster=auto -o ControlPersist=60s
site.yml
---
# Master playbook — delegates to sub-playbooks

- import_playbook: webservers.yml
- import_playbook: databases.yml
04 — Inventory

Inventory Files

The inventory is a list of your managed hosts, organized into groups. It can be a static file (INI or YAML) or a dynamic script/plugin that queries a cloud provider.

inventory/production
# Ungrouped hosts
mail.example.com

# Group: webservers
[webservers]
web-01.example.com
web-02.example.com  ansible_port=2222

# Group: databases
[databases]
db-01.example.com
db-02.example.com

# Group of groups
[production:children]
webservers
databases

# Variables for a group
[webservers:vars]
http_port=80
max_connections=100
inventory/hosts.yml
---
all:
  children:
    webservers:
      hosts:
        web-01:
          ansible_host: 192.168.1.10
          ansible_user: ubuntu
        web-02:
          ansible_host: 192.168.1.11
      vars:
        http_port: 80
    databases:
      hosts:
        db-01:
          ansible_host: 192.168.1.20

Dynamic inventory plugins query live sources — AWS, GCP, Azure, vSphere, etc.

inventory/aws_ec2.yml
---
plugin: amazon.aws.aws_ec2
regions:
  - eu-west-1
filters:
  instance-state-name: running
  tag:Environment: production
keyed_groups:
  - key: tags.Role
    prefix: role
terminal
# Install the collection first
ansible-galaxy collection install amazon.aws
# List discovered hosts
ansible-inventory -i inventory/aws_ec2.yml --list
05 — Playbooks

Writing Playbooks

A playbook is a YAML file containing one or more plays. Each play maps a group of hosts to a set of tasks.

webservers.yml
---
- name: Configure web servers
  hosts: webservers
  become: true          # run as sudo

  vars:
    app_port: 8080
    app_name: myapp

  pre_tasks:
    - name: Update apt cache
      ansible.builtin.apt:
        update_cache: true
        cache_valid_time: 3600

  tasks:
    - name: Install nginx
      ansible.builtin.package:
        name: nginx
        state: present

    - name: Deploy config from template
      ansible.builtin.template:
        src: nginx.conf.j2
        dest: /etc/nginx/nginx.conf
        mode: '0644'
      notify: Reload nginx   # triggers handler

    - name: Ensure nginx is running
      ansible.builtin.service:
        name: nginx
        state: started
        enabled: true

  handlers:
    - name: Reload nginx
      ansible.builtin.service:
        name: nginx
        state: reloaded

pre_tasks / post_tasks

Run before or after the main tasks block, regardless of roles.

handlers

Tasks that run only when notified by another task, and only once at the end of the play.

notify

A task triggers a handler by name. The handler runs at the end if the task reports changed.

06 — Roles

Roles

Roles are a way to package and reuse automation. Each role is a directory with a fixed layout. Tasks, handlers, variables, templates, and files live in known subdirectories — Ansible loads them automatically.

roles/nginx/tasks/main.yml
---
- name: Install nginx
  ansible.builtin.package:
    name: nginx
    state: present

- name: Deploy vhost config
  ansible.builtin.template:
    src: vhost.conf.j2
    dest: /etc/nginx/sites-enabled/{{ vhost_name }}.conf
  notify: Reload nginx
roles/nginx/defaults/main.yml
---
nginx_port: 80
vhost_name: default
nginx_worker_processes: auto

Using a role in a playbook

webservers.yml
---
- hosts: webservers
  roles:
    - common
    - role: nginx
      vars:
        nginx_port: 443
        vhost_name: myapp

Installing roles from Ansible Galaxy

requirements.yml
---
roles:
  - name: geerlingguy.docker
  - name: geerlingguy.nginx

collections:
  - name: amazon.aws
    version: ">=6.0.0"
terminal
ansible-galaxy install -r requirements.yml

08 — Modules

Essential Modules

Modules are the units of work Ansible executes. Always prefer the FQCN (Fully Qualified Collection Name) format like ansible.builtin.copy.

apt / yum / package
# Generic (auto-detects OS)
- ansible.builtin.package:
    name: [git, curl, vim]
    state: present  # present | absent | latest

# Debian/Ubuntu specific
- ansible.builtin.apt:
    name: nginx
    state: latest
    update_cache: true
file / copy / template / lineinfile
# Create directory
- ansible.builtin.file:
    path: /opt/myapp
    state: directory
    owner: www-data
    mode: '0755'

# Copy static file
- ansible.builtin.copy:
    src: files/app.conf
    dest: /etc/app/app.conf

# Render Jinja2 template
- ansible.builtin.template:
    src: nginx.conf.j2
    dest: /etc/nginx/nginx.conf

# Ensure a line exists in a file
- ansible.builtin.lineinfile:
    path: /etc/ssh/sshd_config
    regexp: '^PermitRootLogin'
    line: 'PermitRootLogin no'
service / systemd
- ansible.builtin.systemd:
    name: nginx
    state: started   # started|stopped|restarted|reloaded
    enabled: true
    daemon_reload: true
command / shell / raw
# command — no shell, safer, no pipes
- ansible.builtin.command:
    cmd: systemctl restart nginx
    creates: /etc/nginx/nginx.conf  # skip if file exists

# shell — full shell, pipes/redirects work
- ansible.builtin.shell: cat /etc/passwd | grep ubuntu
  register: shell_output

# Use the result
- ansible.builtin.debug:
    msg: "{{ shell_output.stdout }}"
user / group / authorized_key
- ansible.builtin.user:
    name: deploy
    groups: [www-data, sudo]
    shell: /bin/bash
    create_home: true

- ansible.posix.authorized_key:
    user: deploy
    key: "{{ lookup('file', '~/.ssh/id_rsa.pub') }}"
pip
- ansible.builtin.pip:
    name: [flask, gunicorn]
    virtualenv: /opt/myapp/venv
    state: present
